# PebblesDB: Building Key-Value Stores using Fragmented Log-Structured Merge Trees

## 背景和动机

LSM-Tree 的写放大问题

  ┌─────────────────────────────────────────────────────────────┐
  │                    LSM-Tree 写放大问题                      │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  问题根源: LSM 要求每层 sstable 的 key 范围不重叠           │
  │  ├── 新数据写入时需要与现有数据合并排序                     │
  │  ├── 同一数据在 Compaction 过程中被反复读写                 │
  │  └── 写放大可达 12× - 50×                                   │
  │                                                             │
  │  传统 LSM Compaction 示例:                                  │
  │  ┌──────----───┐     ┌────---─────┐     ┌──────---───┐      │
  │  │ L0: {1,100} │  →  │ L1: {1,50} │  →  │ L2: {1,25} │      │
  │  │             │     │ {100,200}  │     │ {25,100}   │      │
  │  └────────----─┘     └───---──────┘     │ {100,400}  │      │
  │                                         └─---────────┘      │
  │  数据 {1,100} 被重写了 3 次！                               │
  │                                                             │
  │  后果:                                                      │
  │  ├── SSD 寿命缩短                                           │
  │  ├── 存储成本增加                                           │
  │  └── 写入吞吐量降低 (RocksDB 写入吞吐仅为读取的 10%)        │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

## 问题

LSM-Tree 有按层合并，还有按级合并，并不是所有类型的 SSTable 的 key 范围都不重叠

所以 PebblesDB 的主要设计思想是在允许多个 sstable key 重叠情况下，保证查询的性能，做到读，写两端的性能都很好。

使用 Guards 机制，受 SkipList 启发

  ┌─────────────────────────────────────────────────────────────┐
  │                    FLSM Guards 机制                         │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  Guard: 从插入的 key 中随机选择的键，用于划分 key 空间      │
  │                                                             │
  │  关键属性:                                                  │
  │  ├── Guard 之间 key 范围不重叠 (disjoint)                   │
  │  ├── 每个 Guard 可以附加多个 sstables (允许重叠)            │
  │  ├── 高层 Guard 包含低层所有 Guard (类似 Skip List)         │
  │  └── 最底层有最多的 Guards                                  │
  │                                                             │
  │  Guard 选择概率 (Guard Probability):                        │
  │  ├── Level 1: 最低概率 (最少 Guards)                        │
  │  ├── Level i+1: 概率 > Level i (更多 Guards)                │
  │  └── 具体: 若概率为 1/10，每 10 个 key 选 1 个作为 Guard    │
  │                                                             │
  │  示例布局:                                                  │
  │                                                             │
  │  Level 3:  [Guard:5]────[Guard:100]────[Guard:375]────[Guard:1023] │
  │              │            │              │               │         │
  │  Level 2:    [G:5]────────[G:100]────────[G:375]                   │
  │              │            │              │                         │
  │  Level 1:    [G:5]────────[G:100]                                  │
  │              │                                                     │
  │  Level 0:    (无 Guards，新 sstables 直接附加)                     │
  │                                                                    │
  │  每个 Guard 下的 sstables (允许重叠):                              │
  │  Guard:5 可能有: {1,20}, {2,35}, {5,40}  (都包含 key 5)            │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

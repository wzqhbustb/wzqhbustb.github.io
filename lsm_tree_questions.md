  │                                                             │

  │  ├── 原因: 查找需要遍历多层 SSTable                         │

  │                                                             │

SSD 与 HDD 的本质差别：

  ┌─────────────────────────────────────────────────────────────┐
  │              HDD vs SSD 性能特征对比                        │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  HDD (机械硬盘):                                            │
  │  ├── 随机 I/O 比顺序 I/O 慢 100×+                           │
  │  └── LSM-Tree 的 trade-off 合理: 用顺序写入换取查询性能     │
  │                                                             │
  │  SSD (固态硬盘):                                            │
  │  ├── 随机读取性能接近顺序读取 (单线程 50%，多线程可持平)    │
  │  ├── 随机写入仍有开销 (擦除-写入周期)                       │
  │  └── 高 I/O 放大浪费带宽且缩短 SSD 寿命                     │
  │                                                             │
  │  结论: LSM-Tree 的设计假设在 SSD 上不再完全适用             │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

## 关键挑战与解决方案

1. 键值分离后，范围查询需要大量随机读取 vLog，性能下降

解决方案：利用 SSD 的并行随机读取能力

  ┌─────────────────────────────────────────────────────────────┐
  │               WiscKey 范围查询优化                          │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  预取 (Prefetching) 机制:                                   │
  │                                                             │
  │  1. 从 LSM-tree 顺序读取一批 keys                           │
  │  2. 提取对应的 value addresses                              │
  │  3. 32 个线程并行随机读取 vLog                              │
  │  4. 利用 SSD 内部并行性达到接近顺序读的带宽                 │
  │                                                             │
  │  性能:                                                      │
  │  ├── 小 value (64B): 比 LevelDB 慢 12× (设备限制)           │
  │  ├── 大 value (4KB+): 比 LevelDB 快 8.4×                    │
  │  └── 顺序写入的数据: 两者性能接近                           │
  │                                                             │
  │  优化: 对于小 value 场景，可进行 log reorganization (排序)  │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

2. 垃圾回收机制

  ┌─────────────────────────────────────────────────────────────┐
  │              WiscKey 垃圾回收机制                           │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  回收过程:                                                  │
  │  1. 从 Tail 开始顺序读取 chunk                              │
  │  2. 对每个 Key-Value 对:                                    │
  │     - 在 LSM-tree 中查找该 Key                              │
  │     - 检查 LSM-tree 中的地址是否匹配当前 vLog 位置          │
  │     - 如果匹配: 有效数据，追加写入 Head                     │
  │     - 如果不匹配: 无效数据，跳过                            │
  │  3. 更新 Tail 指针                                          │
  │  4. 使用 fallocate() punch hole 释放空间                    │
  │                                                             │
  │  一致性保证:                                                │
  │  ├── 先 fsync() vLog 追加的数据                             │
  │  ├── 再同步更新 LSM-tree 中的 Tail 指针                     │
  │  └── 崩溃后: Tail 之前的都已持久化，之后的可能丢失          │
  │                                                             │
  │  开销:                                                      │
  │  ├── 100% 无效数据: 吞吐量下降 < 10%                        │
  │  ├── 25% 无效数据: 吞吐量下降约 35%                         │
  │  └── 可通过调节 GC 频率平衡                                 │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

3. 崩溃一致性，键值分离后保证崩溃恢复时的数据一致性

解决方案，利用现代文件系统的追加原子性

  ┌─────────────────────────────────────────────────────────────┐
  │              WiscKey 崩溃一致性保证                         │
  ├-────────────────────────────────────────────────────────────┤
  │                                                             │
  │  文件系统保证:                                              │
  │  ├── 追加操作的原子性: 崩溃后，文件只包含完整的追加前缀     │
  │  └── 不会出现: 部分写入、乱序写入、垃圾数据                 │
  │                                                             │
  │  WiscKey 策略:                                              │
  │  ├── Value 先追加到 vLog                                    │
  │  ├── 再写入 Key + Address 到 LSM-tree                       │
  │  └── 恢复时: 从 LSM-tree 记录的 Head 开始扫描 vLog          │
  │                                                             │
  │  异常处理:                                                  │
  │  ├── Key 在 LSM-tree 中，但 Value 丢失: 删除 Key            │
  │  ├── Value 存在，但 Key 丢失: 会被 GC 回收                  │
  │  └── 查询时验证: 检查地址范围 + Key 匹配                    │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

  七、论文贡献总结

  1. 问题识别: 指出 LSM-Tree 的高 I/O 放大问题在 SSD 上尤为严重
  2. 核心创新: 提出键值分离架构，大幅降低 Compaction 开销
  3. 工程实现: 解决范围查询、垃圾回收、崩溃一致性三大挑战
  4. 实验证明: 在多种工作负载下性能显著优于 LevelDB/RocksDB
  5. 开源代码: 基于 LevelDB 1.18 实现，可复现

  WiscKey 证明了针对 SSD 特性重新设计存储系统的巨大潜力，是 LSM-Tree 优化领域的重要里程碑。

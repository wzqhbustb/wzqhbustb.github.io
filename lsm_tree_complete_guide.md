## 一、概述

### 1.1 什么是 LSM-Tree

LSM-Tree (Log-Structured Merge-Tree) 是一种**写优化**的持久化数据结构，专为高写入吞吐量场景设计。它通过将随机写转换为顺序写，显著提升了写入性能。

### 1.2 核心设计思想

LSM-Tree 的设计围绕四个核心原则展开：

**1. 延迟与批量处理 (Defer and Batch)**
- 写入操作首先记录在内存结构（MemTable）中，避免立即磁盘 I/O
- 当 MemTable 达到一定阈值后，批量刷写到磁盘形成 SSTable
- 后台通过 Compaction 合并维护数据有序性
- 这种批量处理摊销了单次写入的磁盘开销

**2. 顺序写优化**
- SSTable 一旦写入不可修改，只允许追加新文件
- 所有磁盘操作都是顺序写，充分利用磁盘带宽（HDD 顺序写可达 100MB/s+，随机写仅 1MB/s）
- 牺牲部分读性能（需要检查多层）换取极致写性能
- 通过 Bloom Filter、缓存等技术补偿读性能

**3. 多级分层结构**
- L0（Level 0）：内存 MemTable + 最近刷盘的不可变文件（允许重叠）
- L1-LN（Level 1-N）：磁盘文件，每层大小按固定比例（通常是 10 倍）递增
- 数据从上层向下层流动，越冷的数据位于越深层
- 上层数据量小、查询快；下层数据量大、查询慢但存储效率高

**4. 空间换时间**
- 允许数据冗余：同一 key 的多个版本可能共存于不同层
- 用写放大（重写数据）换取顺序 I/O 的高吞吐
- 用空间放大（暂时保留无效数据）换取删除操作的高效（只需标记 Tombstone）

## 二、LSM-Tree 理论基础 (1996)

### 2.1 论文核心贡献

**论文**: *The Log-Structured Merge-Tree (LSM-Tree)*  
**作者**: Patrick O'Neil, Edward Cheng, Dieter Gawlick, Elizabeth O'Neil  
**发表**: Acta Informatica, 1996

#### 解决的问题

传统 B-Tree 在高写入场景下的问题：
- **随机写入**: 每次插入需要随机磁盘 I/O
- **索引维护成本**: 实时索引使事务 I/O 成本翻倍
- **写放大**: B-Tree 更新可能需要多次磁盘写

#### 核心创新

**双组件架构**: LSM-Tree 将存储分为内存组件 C0 和磁盘组件 C1。
- C0 完全驻留内存，支持快速插入和查询
- C1 驻留磁盘，以多页块形式组织，支持顺序读写
- 数据通过 Rolling Merge 从 C0 渐进迁移到 C1

**Rolling Merge 机制**: 一种类似归并排序的渐进式数据迁移过程。
- 批量读取 C1 的多页块到内存缓冲区
- 从 C0 读取一段连续条目，与 C1 块内数据合并
- 创建新的 C1 块并写入新位置（不覆盖旧块）
- 循环往复，实现数据的持续下刷

**成本模型**: LSM-Tree 相比 B-Tree 的成本优势来源于磁盘 I/O 模式差异。
- B-Tree：随机页 I/O，每次更新可能触发多次磁盘寻道
- LSM：多页块顺序 I/O，批量写入摊销寻道成本
- 成本比 ≈ (顺序 I/O 成本/随机 I/O 成本) × (1/合并批次大小)

**双组件架构示意图**:

                    LSM-Tree 双组件架构
    ┌─────────────────────────────────────────────────────┐
    │                                                     │
    │    C0 组件 (内存)              C1 组件 (磁盘)          │
    │    ┌──────────────┐            ┌──────────────┐     │
    │    │  索引结构      │            │  多页块 1    │     │
    │    │  (B-Tree)    │            ├──────────────┤     │
    │    ├──────────────┤  Rolling   │  多页块 2    │     │
    │    │ Key1: Val1   │  Merge     ├──────────────┤     │
    │    │ Key2: Val2   │ ═════════▶ │  多页块 3    │     │
    │    │ Key3: Val3   │            ├──────────────┤     │
    │    │    ...       │            │     ...      │     │
    │    └──────────────┘            └──────────────┘     │
    │                                                     │
    │    特点：                          特点：            │
    │    - 完全内存驻留                   - 磁盘顺序存储   │
    │    - 支持快速读写                   - 追加写优化     │
    │    - 容量受限                       - 大容量         │
    │                                                     │
    └─────────────────────────────────────────────────────┘

### 2.2 滚动归并 (Rolling Merge) 机制详解

Rolling Merge 是 LSM-Tree 最核心的机制，负责将数据从内存组件迁移到磁盘组件。

**Rolling Merge 过程示意图**:

    步骤 1: 读取 C1 多页块              步骤 2: 合并 C0 条目
    ┌─────────────────────┐            ┌─────────────────────┐
    │ C0: [a,b,c,d,e]     │            │ C0: [d,e]           │
    │      ↑              │            │       (已合并)      │
    │  合并游标           │            │                     │
    │                     │            │ C1 Buffer:          │
    │ C1 Buffer:          │            │ [a,b,c] + [d,e]     │
    │ [块1: x,y,z]        │ ════════▶  │ = [a,b,c,d,e]       │
    │                     │            │                     │
    └─────────────────────┘            └─────────────────────┘

    步骤 3: 写入新 C1 块                步骤 4: 推进游标
    ┌─────────────────────┐            ┌─────────────────────┐
    │ C0: [d,e,f,g]       │            │ C0: [f,g,h,i,j]     │
    │       (下一批)      │            │          ↑          │
    │                     │            │      新游标位置     │
    │ 新 C1 块:           │ ════════▶  │                     │
    │ [a,b,c,d,e]         │            │ C1 Buffer:          │
    │ (写入磁盘新位置)    │            │ [块2: m,n,o]        │
    └─────────────────────┘            └─────────────────────┘

**工作流程**:

1. **初始化阶段**
   - 确定 C1 中待合并的多页块范围（通常是连续的磁盘块）
   - 将这些块读入内存缓冲区

2. **合并阶段**
   - 从 C0 读取一段连续条目（称为合并游标）
   - 将 C0 条目与 C1 块内已有数据按 key 排序合并
   - 处理更新（覆盖旧值）和删除（生成 Tombstone）

3. **写入阶段**
   - 将合并后的数据写入 C1 的新位置
   - 采用追加写而非覆盖写，保证崩溃安全
   - 更新索引指向新块

4. **推进阶段**
   - 移动合并游标到 C0 的下一段
   - 循环执行直到 C0 数据全部迁移

**关键特性**:
- **渐进性**: 不是一次性全量合并，而是分批次进行，控制单次 I/O 开销
- **并发安全**: 通过"填充块"和"清空块"设计，支持并发读写
- **崩溃恢复**: 利用 WAL 和检查点，失败后可从中断处恢复

**与 LevelDB Compaction 的区别**:
- Rolling Merge: 持续的、按固定节奏进行的合并
- LevelDB Compaction: 按需触发、批量处理的后台任务

### 2.3 多组件架构

当双组件（C0+C1）的内存成本过高时，论文提出扩展到多组件架构。

**架构形式**:

              LSM-Tree 多组件架构
    ┌────────────────────────────────────────┐
    │                                        │
    │   C0 (内存)                            │
    │   ┌────────────────┐                   │
    │   │ Key range:     │ ← 新写入         │
    │   │ [latest keys]  │                   │
    │   └───────┬────────┘                   │
    │           │ Rolling Merge              │
    │           ▼                            │
    │   C1 (磁盘)                            │
    │   ┌────────────────┐  大小: 10× C0    │
    │   │ Key range:     │ ← 较旧数据       │
    │   │ [older keys]   │                   │
    │   └───────┬────────┘                   │
    │           │ Rolling Merge              │
    │           ▼                            │
    │   C2 (磁盘)                            │
    │   ┌────────────────┐  大小: 10× C1    │
    │   │ Key range:     │ ← 更旧数据       │
    │   │ [oldest keys]  │                   │
    │   └───────┬────────┘                   │
    │           │                            │
    │           ▼                            │
    │   C3, C4, ... (更多磁盘层)             │
    │                                        │
    │   每层容量递增 (通常 10 倍)            │
    │   数据越冷，所在层越深                 │
    │                                        │
    └────────────────────────────────────────┘

**组件大小设计**:
- 最优大小比例: r = Si/Si-1 = (SK/S0)^(1/K)
- Si: 第 i 个组件的大小
- S0: 最小组件（通常是 C0）
- SK: 最大组件
- K: 组件数量

**实际经验**:
- 通常使用 3 个组件（C0, C1, C2）即可满足大部分场景
- 超过 3 个组件的收益递减，但管理复杂度显著增加
- C0 完全内存驻留，C1 和 C2 磁盘驻留

**合并策略**:
- C0→C1: 最频繁的合并，保持 C0 大小可控
- C1→C2: 较少频率，C1 达到一定阈值后触发
- 异步执行：各层合并可并行进行

这种多组件设计通过增加存储层级，在保持写入性能的同时，显著降低了内存成本。实际系统中（如 LevelDB、RocksDB）采用的多级 Compaction 策略正是这一思想的工程实现。

### 2.4 操作语义

#### 删除操作

LSM-Tree 的删除是**逻辑删除**而非物理删除：

**Tombstone 机制示意图**:

    步骤 1: 在 C0 插入 Tombstone
    ┌─────────────────────────────────────────┐
    │  C0 (内存)                              │
    │  ┌──────────────────────────────────┐   │
    │  │ key=X: [Tombstone]  ← 新写入     │   │
    │  │ key=A: value1                    │   │
    │  │ key=B: value2                    │   │
    │  └──────────────────────────────────┘   │
    │                                         │
    │  C1 (磁盘)                              │
    │  ┌──────────────────────────────────┐   │
    │  │ key=X: old_value  ← 待清理       │   │
    │  │ key=C: value3                    │   │
    │  └──────────────────────────────────┘   │
    └─────────────────────────────────────────┘

    步骤 2: Rolling Merge 传播
    ┌─────────────────────────────────────────┐
    │  C0 (内存)                              │
    │  ┌──────────────────────────────────┐   │
    │  │ key=A: value1                    │   │
    │  │ key=B: value2                    │   │
    │  └──────────────────────────────────┘   │
    │           │                             │
    │           ▼ Rolling Merge               │
    │                                         │
    │  C1 (磁盘)                              │
    │  ┌──────────────────────────────────┐   │
    │  │ key=X: [Tombstone]  ← 传播到 C1  │   │
    │  │ key=X: old_value    ← 被标记清理 │   │
    │  │ key=C: value3                    │   │
    │  └──────────────────────────────────┘   │
    └─────────────────────────────────────────┘

    步骤 3: 合并时湮灭
    ┌─────────────────────────────────────────┐
    │  C1 (新块写入)                          │
    │  ┌──────────────────────────────────┐   │
    │  │ key=A: value1                    │   │
    │  │ key=B: value2                    │   │
    │  │ key=C: value3                    │   │
    │  │ (key=X 已清理，无记录)            │   │
    │  └──────────────────────────────────┘   │
    └─────────────────────────────────────────┘

**机制说明**:

1. **Tombstone 生成**
   - 在 C0 中插入特殊的"删除标记"条目（Tombstone）
   - 标记包含被删除 key 和删除时间戳
   - 原 value 位置不再重要，只需标记 key 被删除

2. **删除传播**
   - Tombstone 随 Rolling Merge 从 C0 传播到 C1、C2...
   - 在合并过程中，Tombstone 与对应的实际条目相遇时"湮灭"
   - 最终被完全清理，释放空间

3. **查询处理**
   - 查询时按从新到旧的顺序扫描各组件
   - 遇到 Tombstone 立即停止搜索，返回 key 不存在
   - 先遇到实际条目则返回该值（表示在删除前写入的版本）

**优势**: 删除操作只需内存写入，无需磁盘随机访问，性能极高。

#### 谓词删除

支持基于条件的批量删除（如"删除 20 天前的数据"）：
- 在 Rolling Merge 过程中检查每个条目
- 符合条件的条目直接丢弃，不写入新块
- 无需单独扫描和标记，利用已有合并流程完成清理

**应用场景**: 时序数据过期、日志清理、隐私数据删除等。

#### 并发控制机制

**组件级锁**:

                 组件级读写锁机制

    │                                        │
    │   读操作 (多个并发)                     │
    │   ┌─────┐ ┌─────┐ ┌─────┐             │
    │   │ R1  │ │ R2  │ │ R3  │             │

    │      └───────┼───────┘                │
    │              ▼                         │
    │   C0: 读锁 (共享) ───────────────┐     │
    │   ┌──────────────────────────────┴─┐   │
    │   │         C0 内存组件            │   │
    │   └────────────────────────────────┘   │
    │                                        │
    │   写操作 (独占)                        │
    │   ┌─────┐                             │
    │   │  W  │ ──▶ C0: 写锁 (独占)         │
    │   └─────┘                             │
    │                                        │
    │   Rolling Merge (需两层锁)             │
    │   ┌─────┐                             │
    │   │ RM  │ ──▶ 同时持有 C0+C1 写锁      │
    │   └─────┘                             │
    │                                        │
    └────────────────────────────────────────┘

**游标交叉处理**:

                 填充块与清空块设计
    ┌────────────────────────────────────────┐
    │                                        │
    │   Rolling Merge 游标位置               │
    │   C0: [a,b,c,d,e,f,g,h]                │
    │            ↑                           │
    │         游标 (已扫描 c)                │
    │                                        │
    │   并发写入 key=c (新数据)              │
    │   ┌────────────────┐                   │
    │   │ 填充块 (Buffer) │ ◀── 暂存新写入    │
    │   │ key=c: new_val  │                   │
    │   └────────────────┘                   │
    │                                        │
    │   C1 待合并块                          │
    │   ┌────────────────┐                   │
    │   │ 清空块 (Locked) │ ◀── 原数据        │
    │   │ [x,y,z,c_old]   │                   │
    │   └────────────────┘                   │
    │                                        │
    │   合并结果                             │
    │   ┌────────────────┐                   │
    │   │ [x,y,z,c_new]   │ ◀── 填充块数据    │
    │   └────────────────┘    合并入结果      │
    │                                        │
    └────────────────────────────────────────┘

- Rolling Merge 使用游标遍历 C0 和 C1
- 并发写可能插入到游标已扫描过的位置
- 解决方案："填充块"与"清空块"设计
  - 合并前锁定 C1 的待处理块（清空块）
  - 并发写暂存到特殊区域（填充块）
  - 合并完成后，填充块数据合并到结果

    │        │                              │
    │   ┌────▼──────────────┐               │
    │   │ 检查点文件         │               │
    │   │ C0 状态快照       │               │
    │   │ C1 边界 key       │               │
    │   └───────────────────┘               │
    │                                        │
    │   崩溃后恢复：                          │
    │   1. 加载检查点 (t=N 状态)              │
    │   2. 重放 WAL (W3, W4, W5...)           │
    │   3. 恢复到最新状态                     │
    │                                        │
    └────────────────────────────────────────┘

**检查点 (Checkpoint)**:
- 定期将 C0 的全部或部分内容刷写到磁盘
- 记录当前各组件状态（大小、边界 key 等）
- 崩溃后从检查点恢复，减少需重放的 WAL 量

**WAL (Write-Ahead Log)**:
- 每次写操作先记录到顺序日志
- 日志记录包含 key、value、操作类型、时间戳
- 崩溃后扫描 WAL 重放未持久化的操作

**恢复流程**:
1. 加载最近的检查点，恢复 C0 和 C1 状态
2. 扫描 WAL，重放检查点之后的所有操作
3. 验证各组件一致性（校验和、大小等）
4. 恢复完成，系统可用

### 3.1 架构总览

LevelDB 采用经典的 LSM-Tree 架构，整体分为内存组件和磁盘组件两大部分。

**层级大小配置**:

| Level | 目标大小 | 文件大小 | 特点 |
|-------|---------|---------|------|
| **Level 0** | 4 MB | ~2 MB | 最多 4 个文件，文件间 key 可能重叠 |
| **Level 1** | 10 MB | ~2 MB | 文件间 key 不重叠 |
| **Level 2** | 100 MB | ~2 MB | 大小是 L1 的 10 倍 |
| **Level 3** | 1 GB | ~2 MB | 依此类推... |
| **Level 4** | 10 GB | ~2 MB | |
| **Level 5** | 100 GB | ~2 MB | |
| **Level 6** | 1 TB | ~2 MB | |

**重要阈值参数**:
- **kL0_SlowdownWritesTrigger = 8**: L0 文件数达到 8 时写入减速
- **kL0_StopWritesTrigger = 12**: L0 文件数达到 12 时停止写入
- **kTargetFileSize = 2 MB**: SSTable 目标文件大小

**写入路径**:
1. 用户调用 `Put(key, value)`
2. 首先顺序写入 WAL（Write-Ahead Log），确保持久性
3. 然后插入内存中的 MemTable（SkipList 实现）
4. 返回写入成功（延迟刷盘）

**MemTable 刷盘（Minor Compaction）**:
- 当 MemTable 达到阈值（默认 4MB），转为 Immutable MemTable
- 后台线程将 Immutable MemTable 写入磁盘成为 Level 0 SSTable
- 写入完成后释放 Immutable MemTable，重置 WAL

**后台合并（Major Compaction）**:
- Level 0 SSTables 之间可能有 key 范围重叠
- 当 Level 0 文件数超过阈值，或下层大小超过限制时触发
- 将 Level N 的 SSTable 与 Level N+1 重叠的文件合并
- 合并后生成新的有序 SSTables 放入下层

**读取路径**:
1. 首先在 MemTable 中查找
2. 然后在 Immutable MemTable 中查找
3. 按从新到旧的顺序扫描各层 SSTables
4. 每层内先通过 Bloom Filter 快速排除
5. 再通过索引定位到具体 Data Block
6. 在 Data Block 内二分查找目标 key

### 3.2 核心数据结构

#### MemTable: SkipList

LevelDB 选择 SkipList（跳表）而非 B-Tree 作为 MemTable 实现：

**SkipList 结构示意图**:

                    SkipList 多层索引结构
    ┌────────────────────────────────────────────────────────────┐
    │                                                             │
    │  Level 3 (最高层):  head ──────────────────────▶ [70]      │
    │                      │                                      │
    │  Level 2:           head ─────▶ [30] ─────────▶ [70]      │
    │                      │          │               │          │
    │  Level 1:           head ─────▶ [30] ──▶ [50] ─▶ [70]      │
    │                      │          │          │      │        │
    │  Level 0 (底层):    head ──▶ [10]─▶ [30]─▶ [50]─▶ [70]─▶ [90] │
    │                                data   data   data   data   │
    │                                                             │
    │  搜索 [50]: 从 Level 3 开始，逐层下降                        │
    │  - L3: head ──▶ 70 (>50), 下降到 L2                          │
    │  - L2: head ──▶ 30 (<50), 前进 ──▶ 70 (>50), 下降到 L1       │
    │  - L1: 30 ──▶ 50 (命中!)                                    │
    │                                                             │
    │  时间复杂度: O(log n)，平均层数 ≈ log₁/ₚ(n)                  │
    │                                                             │
    └────────────────────────────────────────────────────────────┘

**SkipList 核心特性**:

- **实现简单**: 约 400 行代码，无需复杂的平衡操作
- **并发友好**: 支持无锁读取（Lock-free read），通过原子操作实现
- **内存高效**: 使用 Arena 分配器批量申请内存，减少碎片
- **范围查询**: 有序结构，支持高效的范围扫描

**节点高度生成**:
- 每个节点的高度随机生成，概率 p = 0.25
- 高度为 k 的概率：p^(k-1) * (1-p)
- 期望高度：1/(1-p) = 1.33（平均每个节点约 1.33 个指针）

**时间复杂度**:
- 查找/插入/删除：O(log n)
- 空间复杂度：O(n)，每个节点平均 1/(1-p) 个指针

**与 B-Tree 对比**:

| 特性 | B-Tree | SkipList | LevelDB 收益 |
|------|--------|----------|--------------|
| 实现代码量 | ~2000+ 行 | **~400 行** | 维护成本降低 80% |
| 并发读 | 需要锁 | **无锁 (Lock-free)** | 读性能提升 2-3× |
| 内存分配 | 频繁分配/释放 | **Arena 批量分配** | 内存碎片减少 |
| 实现复杂度 | 高（需平衡） | **低（概率平衡）** | Bug 率降低 |

#### SSTable 文件格式详解

SSTable (Sorted String Table) 是 LevelDB 的核心磁盘文件格式，设计目标是**高效存储有序键值对**并**支持快速查询**。

##### 整体结构

SSTable 文件从上到下分为四个区域：

**数据区 (Data Blocks)**:
- 存储实际的键值对记录
- 默认每个 Data Block 4KB（压缩前）
- 块内记录按 key 有序排列
- 使用前缀压缩（共享前缀）减少空间

**元数据区 (Meta Blocks)**:
- Filter Block: 存储 Bloom Filter 数据，加速不存在 key 的查询
- Meta Index Block: 指向 Filter Block 的索引

**索引区 (Index Block)**:
- 每个 Data Block 对应一个索引项
- 索引项 key: Data Block 的最大 key
- 索引项 value: Data Block 的偏移量和大小

**尾部 (Footer)**:
- 固定 48 字节
- 包含 Meta Index Block 和 Index Block 的位置信息
- Magic Number 用于文件格式验证

##### Data Block 内部结构

Data Block 是 SSTable 中存储实际键值对的最小单元，默认大小为 **4KB**（压缩后）。

**结构组成**:
1. **键值记录序列**: 实际存储的数据
2. **重启点数组 (Restart Points)**: 用于加速块内查找

**前缀压缩机制**:
- 块内记录按 key 排序，相邻 key 通常有共同前缀
- 每条记录只存储与前一条 key 的差异部分
- 每 16 条记录设置一个重启点（Restart Point），该点存储完整 key
- 查找时从最近的重启点开始解码，减少计算量

**记录格式**:
- Shared Length: 与前一条 key 共享的前缀长度
- Non-shared Length: 本条 key 的非共享部分长度
- Value Length: value 长度
- Key Delta: 非共享的 key 字节
- Value: 实际值

**查找过程**:
1. 在重启点数组中二分查找，确定目标记录所在的范围
2. 从该范围的起始重启点开始顺序解码
3. 比较 key，直到找到或超出范围

**Bloom Filter 参数**:
- **bits_per_key**: 默认 10 bits
- **k (哈希函数个数)**: 根据 bits_per_key 计算，k ≈ 0.693 * bits_per_key ≈ 7
- **误判率**: ~1% (当 bits_per_key = 10)

**误判率计算**:
```
P(false positive) ≈ (1 - e^(-k*n/m))^k

其中:
- n = key 数量
- m = bits 总数
- k = 哈希函数个数
```

| bits_per_key | 误判率 | 内存开销 (每百万 key) |
|--------------|--------|----------------------|
| 5 | 5.0% | 0.625 MB |
| 8 | 1.0% | 1.0 MB |
| 10 | 0.8% | 1.25 MB |
| 15 | 0.05% | 1.875 MB |

**写合并 (Write Batching) 详解**:

写合并是 LevelDB 提升写入吞吐的关键优化。

**工作机制**:
1. **写入队列**: 所有写请求进入 FIFO 队列
2. **组批处理**: 队列头部的写入者负责合并后续多个写入请求
3. **批量 WAL**: 合并后的批量数据一次写入 WAL
4. **批量 MemTable**: 批量数据一次插入 MemTable
5. **通知机制**: 完成后通知所有等待的写入者

**效果量化**:

| 线程数 | 无合并吞吐 | 有合并吞吐 | 提升 |
|--------|-----------|-----------|------|
| 1 线程 | 100 Kops/s | 100 Kops/s | 1× |
| 4 线程 | 150 Kops/s | 380 Kops/s | 2.5× |
| 8 线程 | 180 Kops/s | 720 Kops/s | 4× |
| 16 线程 | 200 Kops/s | 950 Kops/s | 4.75× |

**为什么能提升？**
- 减少 WAL 写入次数（多次随机写 → 一次顺序写）
- 减少锁竞争（串行化处理）
- 提高 CPU 缓存命中率（批量处理）

### 3.4 Compaction 机制详解

Compaction 是 LSM-Tree 的核心机制，负责将数据从上层合并到下层，同时维护有序性。LevelDB 中有两种 Compaction 类型：
- **Minor Compaction**: MemTable → SSTable (Level 0)
- **Major Compaction**: Level N → Level N+1

#### 3.4.1 Minor Compaction 详细流程

Minor Compaction 将内存中的 Immutable MemTable 转换为磁盘上的 Level 0 SSTable。

**触发条件**:
- MemTable 大小达到阈值（默认 4MB）
- 后台线程持续检查，条件满足即触发

**详细步骤**:

**步骤 1: 创建 Immutable MemTable**
- 将当前的 MemTable 标记为 Immutable（不可变）
- 创建新的 MemTable 接收新写入
- 切换 WAL 到新的日志文件

**步骤 2: 智能选择输出 Level**
- 默认输出到 Level 0
- 但 LevelDB 会检查是否可以"跳级"到更高层
- 如果 Immutable MemTable 的 key 范围与 Level 0 及更高层不重叠，可直接放入 L1/L2
- 减少后续 L0→L1 Compaction 的开销

**步骤 3: 写入 SSTable**
- 遍历 Immutable MemTable 的所有 key-value
- 按顺序写入新的 SSTable 文件
- 构建 Data Blocks、Index Block、Filter Block
- 最后写入 Footer

**步骤 4: 更新元数据**
- 将新 SSTable 信息加入 VersionSet
- 记录到 MANIFEST 文件确保持久性
- 删除旧的 WAL 文件
- 释放 Immutable MemTable 内存

**特殊优化 - 跳级选择**:
LevelDB 通过检查 key 范围重叠情况，决定 Immutable MemTable 的输出层级：
- 若与 L0 无重叠，尝试放入 L1
- 若与 L1 也无重叠，且与 L2 重叠不大，可放入 L2
- 以此类推，最高可到 L4
- 这种智能选择显著减少小范围更新的 Compaction 开销

#### 3.4.2 Major Compaction 详细流程

Major Compaction 将数据从上层合并到下层，维护有序性和层级约束。

**触发条件**:
- Level 0 文件数超过阈值（默认 4 个）
- 某层总大小超过目标值（L1: 10MB, L2: 100MB, L3: 1GB...）
- 手动触发（compact_range API）

**Compaction 选择算法**:

**L0→L1 Compaction 特殊性**:
- Level 0 SSTables 之间可能有重叠，必须全部参与合并
- 找出所有与待合并 L0 文件有重叠的 L1 文件
- 多路归并所有重叠文件，生成新的 L1 SSTables

**L1+ Compaction 标准流程**:
- 选择该层中"最拥挤"的 SSTable（文件大小/层大小比例最大）
- 只选择这一个文件，找出 L+1 层中与其重叠的文件
- 合并选中的文件，生成新的 L+1 SSTables
- 删除旧的文件

**Compaction 执行流程**:
1. **输入收集**: 确定所有需要合并的 SSTables
2. **多路归并**: 使用最小堆合并多个有序输入
3. **输出生成**: 按顺序写入新 SSTables，保持目标文件大小（默认 2MB）
4. **元数据更新**: 原子更新 VersionSet，记录到 MANIFEST
5. **旧文件清理**: 删除已合并的旧 SSTables（实际延迟删除，确保无引用后才清理）

**为什么 L0→L1 代价最高**:
- L0 文件重叠导致需要读取更多文件
- L1 大小限制（10MB）导致频繁触发
- 是整个 LSM-Tree 的瓶颈，后续优化（如 Universal Compaction）多针对此

#### 3.4.3 Compaction 中的垃圾回收

Compaction 不仅是合并数据，还负责清理过期数据。

**垃圾回收判断逻辑**:

在 Compaction 过程中，对每个 key 判断是否丢弃：

1. **被新版本隐藏**:
   - 如果同一个 user key 有更新的版本（更高的 sequence number）
   - 且没有 snapshot 引用旧版本
   - 则丢弃旧版本

2. **删除标记清理**:
   - 如果是删除标记 (kTypeDeletion)
   - 且没有 snapshot 引用该标记
   - 且更高层没有该 key 的数据
   - 则可以丢弃该删除标记

**版本可见性规则**:
- 每个 snapshot 记录创建时的 sequence number
- 读取时只能看到 ≤ snapshot sequence number 的数据
- Compaction 时检查 oldest snapshot，旧于它的版本可清理

- 记录所有版本变更（VersionEdit）
- 包括：新增文件、删除文件、比较器变更等

**快照 (Snapshot)**:
- 创建时记录当前 sequence number
- 读取时只读取 ≤ snapshot sequence number 的数据

- 释放后，旧版本可被 Compaction 清理

#### 3.5.1 版本管理流程

1. **正常读写**:
   - 获取 current version 引用 (refs++)
   - 基于该 version 的 SSTable 集合进行读写
   - 完成时释放引用 (refs--)

2. **Compaction 完成**:
   - 创建新 version，包含新的 SSTable 集合
   - 原子切换 current_ 指向新 version
   - 旧 version 引用计数递减

3. **旧版本清理**:
   - 当 version 的 refs == 0 时，可回收
   - 该 version 独有的 SSTable 文件可被删除

### 3.6 读取优化

LevelDB 针对 LSM-Tree 读取需要检查多层的问题，设计了多层优化策略。

**读取放大来源**:
- 最坏情况：需要检查 MemTable + Immutable + L0-L6 共 8 层
- 每层可能需要打开一个 SSTable，读取 Bloom Filter、Index Block、Data Block
- 对于不存在的 key，需要检查所有层才能确认

**优化策略**:

**1. MemTable 优先**
- 最新数据总在 MemTable 中，查找最快
- SkipList O(log n) 查找复杂度

**2. SSTable 缓存 (Table Cache)**
- 缓存已打开的 SSTable 文件描述符和元数据
- 避免频繁打开/关闭文件的系统调用开销

**3. Block Cache**
- LRU 缓存最近读取的 Data Block
- 热点数据直接内存命中，无需磁盘读取
- 默认 8MB，可配置

**4. Bloom Filter 过滤**
- 每个 SSTable 都有对应的 Bloom Filter
- 查询时先检查 Filter，快速排除不含目标 key 的文件
- 对于不存在的 key，减少 90%+ 的磁盘读取

**5. 层级跳过优化**
- 利用 SSTable 文件的有序性，维护每层的 key 范围索引
- 若目标 key < 某层最小 key 或 > 最大 key，直接跳过该层

**6. 预读优化**
- 范围查询时，读取当前 Block 的同时异步预读下一个 Block
- 利用磁盘顺序读的高带宽

**读取性能对比（有无优化）**:

| 场景 | 无优化 | 有优化 | 提升 |
|------|--------|--------|------|
| 点查（存在） | 3-8 ms | 0.1-0.5 ms | 6-30× |
| 点查（不存在） | 10-20 ms | 0.5-1 ms | 10-40× |
| 范围查询 | 线性扫描 | 二分+顺序读 | 10-100× |

## 五、WiscKey 键值分离优化 (2016)

> **论文**: WiscKey: Separating Keys from Values in SSD-Conscious Storage  
> **作者**: Lanyue Lu, Thanumalayan Sankaranarayana Pillai, Andrea C. Arpaci-Dusseau, Remzi H. Arpaci-Dusseau  
> **单位**: University of Wisconsin, Madison  
> **发表**: FAST 2016  
> **核心思想**: 键值分离 - 将值从 LSM-tree 中分离出来，大幅降低 I/O 放大

### 5.1 研究背景与动机

#### 5.1.1 LSM-Tree 的核心问题

LSM-Tree（如 LevelDB、RocksDB）虽然避免了随机写入，但存在严重的 **I/O 放大** 问题：

**写放大 (Write Amplification): 12× - 50×**
- 原因: Compaction 过程中数据被反复读写
- 示例: 1GB 数据写入可能需要 12-50GB 实际磁盘写入

**读放大 (Read Amplification): 3× - 14×**
- 原因: 查找需要遍历多层 SSTable
- 示例: 读取 1KB 数据可能需要读取 3-14KB 实际数据

**空间放大 (Space Amplification)**
- 因无效数据未及时清理，实际占用大于原始数据

#### 5.1.2 SSD 与 HDD 的本质差异

**HDD (机械硬盘)**:
- 随机 I/O 比顺序 I/O 慢 100×+
- LSM-Tree 的 trade-off 合理: 用顺序写入换取查询性能

**SSD (固态硬盘)**:
- 随机读取性能接近顺序读取 (单线程 50%，多线程可持平)
- 随机写入仍有开销 (擦除-写入周期)
- 高 I/O 放大浪费带宽且缩短 SSD 寿命
- 内部并行性: 多个通道/芯片可并行处理随机读取

**结论**: LSM-Tree 的设计假设在 SSD 上不再完全适用

#### 5.1.3 关键观察

1. **Compaction 只关心 Key 的顺序**: 值的排序对范围查询并非必需
2. **Key 通常比 Value 小得多**: 现代 workload 中 key 约 16B，value 100B-4KB+
3. **SSD 并行随机读 ≈ 顺序读**: 这使得键值分离后的范围查询可行

#### 5.2.2 详细数据布局

**LSM-tree 中的记录格式**:
```
Key | Value-Address (vLog offset)
```

**vLog 中的记录格式**:
```
Key | Value | Value-Size (用于 GC 验证)
```

**vLog 管理**:
- **Head 指针** (内存中): 新 Value 追加位置
- **Tail 指针** (持久化): 有效数据起始位置
- Tail 和 Head 指针都存储在 LSM-tree 中: `<"tail", tail-vLog-offset>`, `<"head", head-vLog-offset>`

#### 5.2.3 设计目标

| 目标 | 说明 |
|------|------|
| **低写放大** | 减少不必要的写入，延长 SSD 寿命 |
| **低读放大** | 提高查询吞吐量，提升缓存效率 |
| **SSD 优化** | 匹配 SSD 的并行随机读特性 |
| **功能完整** | 支持范围查询、快照等 LSM-tree 特性 |
| **现实 key-value 大小** | 优化小 key (16B) + 变长 value 场景 |

### 5.3 关键挑战与解决方案

#### 5.3.1 挑战 1: 范围查询 (Range Query) 性能

**问题**: 键值分离后，范围查询需要随机读取 vLog，性能可能下降

**解决方案**: 利用 SSD 并行随机读取 + 预取

**预取 (Prefetching) 机制**:

1. 从 LSM-tree 顺序读取一批 keys (如 1000 个)
2. 提取对应的 value addresses
3. 将 addresses 放入工作队列
4. 32 个后台线程并行随机读取 vLog
5. 利用 SSD 内部并行性达到接近顺序读的带宽

**性能结果**:
- 小 value (64B): 比 LevelDB 慢 12× (设备随机读限制)
- 大 value (4KB+): 比 LevelDB 快 8.4×
- 顺序写入的数据: 两者性能接近 (vLog 中数据已排序)

**进一步优化**: 对小 value 场景可进行 log reorganization (排序)

#### 5.3.2 挑战 2: 垃圾回收 (Garbage Collection)

**问题**: 删除/更新后，vLog 中会产生无效 value，需要回收空间

**解决方案**: 在线轻量级垃圾回收器

**触发条件**:
- 配置为定期运行
- 或空间使用达到阈值
- 支持离线模式用于维护

**回收过程**:

1. 从 Tail 开始顺序读取 chunk (如 16MB)
2. 对每个 Key-Value 对:
   - 在 LSM-tree 中查找该 Key
   - 检查 LSM-tree 中的地址是否匹配当前 vLog 位置
   - 如果匹配: 有效数据，追加写入 Head
   - 如果不匹配: 无效数据，跳过
3. 更新 Tail 指针
4. 使用 fallocate(FALLOC_FL_PUNCH_HOLE) 释放空间

**一致性保证**:
- 先 fsync() vLog 追加的数据
- 再同步更新 LSM-tree 中的 Tail 指针
- 崩溃后: Tail 之前的都已持久化，之后的可能丢失

**性能开销 (随机写入负载)**:
| 无效数据比例 | 吞吐量下降 |
|-------------|-----------|
| 100% | < 10% |
| 50% | ~25% |
| 25% | ~35% |

可通过调节 GC 频率和空间预留平衡

#### 5.3.3 挑战 3: 崩溃一致性 (Crash Consistency)

**问题**: 键值分离后，如何保证崩溃恢复时的数据一致性？

**解决方案**: 利用现代文件系统的追加原子性

**文件系统保证 (ext4, btrfs, xfs)**:
- 追加操作的原子性: 崩溃后，文件只包含完整的追加前缀
- 不会出现: 部分写入、乱序写入、垃圾数据

**WiscKey 写入顺序**:
1. Value 先追加到 vLog
2. fsync() vLog 确保持久化
3. Key + Address 写入 LSM-tree (通过其 WAL 保证持久化)

**崩溃恢复过程**:
- 从 LSM-tree 读取 Head 指针
- 从 Head 开始扫描 vLog 到文件末尾
- 重建 Key → Address 映射
- 重复数据由 LSM-tree 去重机制处理

**异常查询处理**:
- Key 在 LSM-tree 中，但 Value 地址超出范围: 删除 Key
- Key 在 LSM-tree 中，但 Value 中的 Key 不匹配: 删除 Key
- Value 存在但 Key 丢失: 会被后续 GC 回收
- 查询时双重验证: 地址范围 + Key 匹配

**恢复时间**:
- LevelDB: 0.7 秒 (1KB value)
- WiscKey: 2.6 秒 (需要扫描 vLog)
- 可通过更频繁持久化 Head 指针优化

#### 5.3.4 额外优化: 移除 LSM-tree 的 WAL

**观察**:
- LSM-tree 的 WAL 用于恢复 MemTable 数据
- WiscKey 的 vLog 已包含完整的 Key-Value 数据
- 可以通过扫描 vLog 恢复，无需额外 WAL

**优化**:
- 完全移除 LSM-tree 的 log 文件，减少一次写入

**恢复过程**:
- 从 LSM-tree 获取最近的 Head 指针
- 从 Head 扫描 vLog 到末尾
- 重建 LSM-tree 索引

**效果**:
- 对小写入尤其有效，减少 ~50% 的写入量

### 5.6 设计启示与后续影响

#### 5.6.1 核心设计启示

1. **工作负载特征决定存储设计**
   - 观察: 现代 workload 中 key 小 (16B)，value 变化大 (100B-4KB+)
   - 洞察: Compaction 只关心 Key 的顺序
   - 结论: 键值分离可显著降低 Compaction 开销

2. **硬件特性影响架构选择**
   - SSD 随机读接近顺序读 → 键值分离可行
   - SSD 内部并行性高 → 多线程随机读有效
   - SSD 寿命受写入量限制 → 降低写放大必要

3. **Trade-off 重新权衡**
   - 传统 LSM: 顺序写 vs 查询性能
   - WiscKey: 更低写放大 + 利用 SSD 并行性

4. **文件系统特性可利用**
   - 追加操作的原子性简化了崩溃一致性设计

#### 5.6.2 适用场景

**WiscKey 最适合**:
- SSD 存储设备
- Value 大小 >= 1KB
- 写密集型 workload
- 需要范围查询但查询范围不太大的场景

**WiscKey 不适合**:
- HDD 存储 (随机读性能差)
- Value 非常小 (64B 以下)
- 大范围顺序扫描为主的工作负载

## 六、PebblesDB FLSM 优化 (2017)

> **论文**: PebblesDB: Building Key-Value Stores using Fragmented Log-Structured Merge Trees  
> **作者**: Pandian Raju, Rohan Kadekodi, Vijay Chidambaram, Ittai Abraham  
> **单位**: University of Texas at Austin, VMware Research  
> **发表会议**: SOSP 2017 (ACM Symposium on Operating Systems Principles)  
> **核心贡献**: 提出 FLSM (Fragmented Log-Structured Merge Trees) 数据结构，结合 Skip List 和 LSM-Tree，显著降低写放大

### 6.1 研究背景与动机

#### 6.1.1 LSM-Tree 的写放大问题

LSM-Tree（如 LevelDB、RocksDB）虽然避免了随机写入，但存在严重的 **写放大** 问题：

**问题根源**: LSM 要求每层 sstable 的 key 范围不重叠
- 新数据写入时需要与现有数据合并排序
- 同一数据在 Compaction 过程中被反复读写
- 写放大可达 12× - 50×

**传统 LSM Compaction 示例**:
```
L0: {1,100}  →  L1: {1,50}  →  L2: {1,25}
                  {100,200}       {25,100}
                                  {100,400}
```
数据 {1,100} 被重写了 3 次！

**后果**:
- SSD 寿命缩短
- 存储成本增加
- 写入吞吐量降低 (RocksDB 写入吞吐仅为读取的 10%)

#### 6.1.2 传统解决方案的局限

| 方案 | 方法 | 局限 |
|------|------|------|
| **不合并 sstable** | 直接将新 sstable 添加到层 | 读性能急剧下降，需要检查多个 sstable |
| **特殊硬件** | 利用 SSD FTL 特性 (NVMKV) | 依赖特定硬件，通用性差 |
| **牺牲读性能** | LSM-trie, Universal Compaction | 不支持范围查询或读性能差 |
| **键值分离** | WiscKey | 范围查询性能下降，需要 GC |

#### 6.1.3 核心洞察

> **LSM 写放大的根本原因是: 要求每层 sstable 的 key 范围不重叠 (disjoint)**

如果允许同一层有重叠的 sstables，就可以避免在 Compaction 时重写数据。

**挑战**: 如何在允许重叠的情况下保持高效的查询性能？

**解决方案**: 使用 **Guards** (受 Skip List 启发)

### 6.2 FLSM 核心设计

#### 6.2.1 Guards 机制

                    FLSM Guards 机制
    ┌─────────────────────────────────────────────────────────────┐
    │                                                              │
    │  Guard: 从插入的 key 中随机选择的键，用于划分 key 空间          │
    │                                                              │
    │  关键属性:                                                    │
    │  ├── Guard 之间 key 范围不重叠 (disjoint)                      │
    │  ├── 每个 Guard 可以附加多个 sstables (允许重叠)                │
    │  ├── 高层 Guard 包含低层所有 Guard (类似 Skip List)            │
    │  └── 最底层有最多的 Guards                                    │
    │                                                              │
    │  Guard 选择概率 (Guard Probability):                          │
    │  ├── Level 1: 最低概率 (最少 Guards)                          │
    │  ├── Level i+1: 概率 > Level i (更多 Guards)                  │
    │  └── 具体: 若概率为 1/10，每 10 个 key 选 1 个作为 Guard       │
    │                                                              │
    │  示例布局:                                                    │
    │                                                              │
    │  Level 3:  [Guard:5]────[Guard:100]────[Guard:375]────[Guard:1023] │
    │              │            │              │               │    │
    │  Level 2:    [G:5]────────[G:100]────────[G:375]              │
    │              │            │              │                    │
    │  Level 1:    [G:5]────────[G:100]                             │
    │              │                                                 │
    │  Level 0:    (无 Guards，新 sstables 直接附加)                    │
    │                                                              │
    │  每个 Guard 下的 sstables (允许重叠):                          │
    │  Guard:5 可能有: {1,20}, {2,35}, {5,40}  (都包含 key 5)        │
    │                                                              │
    └─────────────────────────────────────────────────────────────┘

#### 6.2.2 FLSM vs LSM 架构对比

                    LSM vs FLSM 架构对比
    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                  │
    │  传统 LSM:                                                       │
    │  ┌─────────────────────────────────────────────────────────┐   │
    │  │ Level 1: [1..100] [200..300] [400..500]                │   │
    │  │          ↑ 每个 key 只在一个 sstable 中                   │   │
    │  │ Level 0: [1..50] [60..80] [200..250]                   │   │
    │  │          ↑ 可以有重叠                                   │   │
    │  └─────────────────────────────────────────────────────────┘   │
    │                                                                  │
    │  FLSM:                                                          │
    │  ┌─────────────────────────────────────────────────────────┐   │
    │  │ Level 1: Guard:50 ────── Guard:200 ────── Guard:400   │   │
    │  │          │                  │                │          │   │
    │  │          ▼                  ▼                ▼          │   │
    │  │        [1,60]             [150,250]        [350,450]   │   │
    │  │        [20,80]            [180,300]        [380,500]   │   │
    │  │        [40,100]           [200,350]        [400,550]   │   │
    │  │        ↑ 同一 Guard 下 sstables 可以重叠                  │   │
    │  └─────────────────────────────────────────────────────────┘   │
    │                                                                  │
    │  关键区别:                                                       │
    │  ├── LSM: 每层 sstables 必须 disjoint                          │
    │  ├── FLSM: Guard 之间 disjoint，Guard 内 sstables 可重叠        │
    │  └── FLSM: 查询时先找 Guard，再检查该 Guard 下所有 sstables      │
    │                                                                  │
    └─────────────────────────────────────────────────────────────────┘

#### 6.2.3 FLSM Compaction 算法

                    FLSM Compaction 过程
    ┌─────────────────────────────────────────────────────────────┐
    │                                                              │
    │  传统 LSM Compaction:                                         │
    │  1. 读取 Level i 的 sstable 和 Level i+1 重叠的 sstables      │
    │  2. 合并排序所有 key-value                                    │
    │  3. 写入新的 sstables 到 Level i+1                            │
    │  4. 删除旧的 sstables                                         │
    │  → 数据被重写！                                               │
    │                                                              │
    │  FLSM Compaction (大部分情况):                                 │
    │  1. 选择 Level i 中一个 Guard (当其 sstables 数量超过阈值)      │
    │  2. 将该 Guard 的所有 sstables 按 Level i+1 的 Guards 分区      │
    │  3. 直接附加 (append) 分区后的 sstables 到 Level i+1 对应 Guard │
    │  4. 删除 Level i 的旧 sstables                                │
    │  → 数据只被分区，不被重写！                                    │
    │                                                              │
    │  示例:                                                        │
    │  Level 1 Guard:5 有 sstable {1, 20, 45, 101, 245}            │
    │  Level 2 Guards: 1, 40, 200                                   │
    │  Compaction 结果:                                             │
    │  ├── Guard:1 获得 {1, 20}                                     │
    │  ├── Guard:40 获得 {45, 101}                                  │
    │  └── Guard:200 获得 {245}                                     │
    │                                                              │
    │  注意: 仅在最高层 (Last Level) 需要重写以控制碎片               │
    │                                                              │
    └─────────────────────────────────────────────────────────────┘

**两种 Compaction 模式**:

| 模式 | 触发条件 | 操作 | 写放大 |
|------|---------|------|--------|
| **Partition Compaction** | 非最高层 | 仅分区，不排序 | 接近 1× |
| **Rewrite Compaction** | 最高层或碎片过多 | 合并排序，重写 | 类似 LSM |

### 6.3 PebblesDB 实现优化

#### 6.3.1 解决 FLSM 的读性能问题

FLSM 的问题：查询需要检查 Guard 下所有 sstables，读性能下降。

**PebblesDB 优化技术**:

| 优化技术 | 说明 | 效果 |
|---------|------|------|
| **SSTable-level Bloom Filter** | 每个 sstable 都有 Bloom Filter | 避免读取不含目标 key 的 sstable |
| **并行 Seek** | 多线程并行搜索 Guard 内多个 sstables | 降低 seek 延迟 |
| **Seek-Based Compaction** | seek 次数超过阈值时触发 Compaction | 减少 Guard 内 sstable 数量 |
| **激进 Compaction** | Level i 大小超过 Level i+1 的 25% 时触发 | 减少需要搜索的层数 |

#### 6.3.2 Guard 的选择算法

**基于哈希的 Guard 选择**:

PebblesDB 使用基于哈希的 Guard 选择算法，避免热点 key 导致 Guards 分布不均。

**核心思想**:
- 对每个 key 计算哈希值
- 检查哈希值的最后 N 位是否全为 1
- Level 越高，N 越小 → 匹配概率越高 → Guards 越多

**概率设计（示例）**:
- Level 1: 需最后 8 位全为 1 → 概率约 1/256（最少 Guards）
- Level 2: 需最后 6 位全为 1 → 概率约 1/64
- Level 3: 需最后 4 位全为 1 → 概率约 1/16（最多 Guards）

这样设计保证高层拥有更多 Guards，形成类似 Skip List 的层级结构。

#### 6.3.3 Guard 的异步插入

Guard 不是同步插入的，而是延迟绑定：

1. 当新 key 通过哈希判断应成为 Guard 时，先加入 "uncommitted_guards" 集合
2. 在后续的 Compaction 过程中，sstables 才会被分区并附加到新 Guard
3. 这种延迟插入避免了同步开销，同时保持 Guard 的层级结构

#### 6.3.4 PebblesDB 操作流程

**Get 操作**:
1. 检查 MemTable
2. 从 Level 0 开始逐层搜索:
   - 二分查找定位 Guard
   - 检查 Guard 下所有 sstables (使用 Bloom Filter 过滤)
   - 找到则返回

**Range Query**:
1. 确定每层涉及的 Guards
2. 对每个 Guard 内的相关 sstables 进行二分查找定位起始 key
3. 使用多路归并 (类似 Merge Sort) 产生有序结果
4. 使用并行读取优化

### 6.5 理论分析

#### 6.5.1 渐进复杂度 (DAM Model)

**假设**:
- 数据项总数: n
- Block 大小: B
- Guard 概率: 1/B (每层 Guards 增加 B 倍)
- 层数 H = log_B(n)

**写成本**:
- 每层只写一次: O(H) = O(log_B(n))
- 最后层重写: O(B) 次
- 总写成本: O((B + log_B(n))/B)

**读成本**:
- 每层二分查找 Guard: O(log(B^H)) = O(H·log B) = O(log n)
- 检查 Guard 内 sstables: 使用 Bloom Filter 后 O(1) (大多数情况)
- 总读成本: O(log n) 内存操作 + O(1) 磁盘读取

#### 6.5.2 FLSM 作为 LSM 的泛化

当 max_sstables_per_guard = 1 时:
- 每个 Guard 只能有 1 个 sstable
- 等价于传统 LSM (每层 sstables disjoint)
- 性能与 LSM 相当

FLSM 优势:
- max_sstables_per_guard > 1: 更低写放大，更高写入吞吐
- 可调参数平衡读写性能

### 6.7 局限性与适用场景

#### 6.7.1 PebblesDB 的局限

1. **顺序写入性能差**
   - LSM 可以直接移动 sstable
   - FLSM 必须分区，产生额外 IO

2. **范围查询开销**
   - 纯 seek 操作慢 30%
   - 大量 next() 可摊平开销

3. **内存消耗**
   - 需要存储 Guard 元数据
   - 更多 Bloom Filters

4. **CPU 开销**
   - Guard 查找
   - 并行搜索协调

#### 6.7.2 适用场景

**最适合**:
- 随机写入主导的工作负载
- 写入吞吐量敏感的场景
- SSD 存储 (延长寿命)
- 写入密集型 NoSQL 应用

**不适合**:
- 纯顺序写入 (日志类)
- 大量小范围查询 (无 next() 摊平)
- 内存受限环境

### 6.8 总结与影响

#### 6.8.1 核心贡献

1. **FLSM 数据结构**: 首次提出 Fragmented LSM，允许每层 sstables 重叠
2. **Guards 机制**: 受 Skip List 启发，高效组织重叠 sstables
3. **PebblesDB 实现**: 证明可以同时实现低写放大、高写入吞吐、可接受的读性能
4. **实际应用**: 成功集成到 MongoDB 和 HyperDex

#### 6.8.2 设计启示

1. 传统 LSM 的 disjoint sstable 约束是写放大的根源
2. 通过 Guards 组织数据可以在允许重叠的同时保持查询效率
3. 写优化和读优化可以平衡，不一定是零和博弈
4. 可调参数 (max_sstables_per_guard) 允许针对不同工作负载优化

#### 6.8.3 关键技术总结

| 技术 | 目的 | 效果 |
|------|------|------|
| Guards | 组织重叠 sstables | 允许不重新排序的 Compaction |
| SSTable Bloom Filters | 加速读取 | 避免读取无关 sstables |
| 并行 Seek | 加速范围查询 | 多线程搜索多个 sstables |
| Seek-Based Compaction | 控制 Guard 大小 | 平衡读写性能 |
| 哈希选择 Guard | 避免倾斜 | 均匀分布 Guards |

#### 6.8.4 后续影响

PebblesDB 的 FLSM 思想影响了后续多个系统：
- 证明了 LSM 可以有更低写放大的实现方式
- Guards 概念被其他系统借鉴
- 启发了更多在 Compaction 算法上的创新
- 展示了 "partial sorting" (部分排序) 的可行性

| **压缩策略** | Leveled | Leveled/Universal/FIFO |

### 7.2 BlobDB: WiscKey 的工业实现

RocksDB 的 BlobDB 是 WiscKey 键值分离思想的工业级实现，将大 value 从 SSTable 中分离存储，显著降低写放大。

#### 7.2.1 架构设计

**键值分离原理**:

传统 LSM:
```
SSTable: [Key1|Value1] [Key2|Value2] ...
         ↓ Compaction 时 Value 被反复重写
```

BlobDB:
```
SSTable: [Key1|BlobRef1] [Key2|BlobRef2] ...
BlobFile: [Value1] [Value2] ... (仅追加，无 Compaction)
```

**核心组件**:

| 组件 | 位置 | 功能 |
|------|------|------|
| **BlobFileBuilder** | `db/blob/blob_file_builder.cc` | 构建 Blob 文件 |
| **BlobFileReader** | `db/blob/blob_file_reader.cc` | 读取 Blob 文件 |
| **BlobIndex** | 存储在 SSTable | 指向 Blob 文件的指针 |
| **BlobGarbageCollector** | `db/blob/blob_garbage_collector.cc` | 垃圾回收 |

#### 7.2.2 Blob 文件格式

```
Blob File Layout
┌─────────────────────────────────────────────────────────────┐
│ Blob Records                                               │
│ ┌───────────────────────────────────────────────────────┐   │
│ │ Key (用于校验) │ Value │ CRC32 │                        │   │
│ └───────────────────────────────────────────────────────┘   │
│ ...                                                         │
├─────────────────────────────────────────────────────────────┤
│ Blob Index (存储在 SSTable 中)                              │
│ ┌───────────────────────────────────────────────────────┐   │
│ │ blob_file_number │ offset │ size │ compression_type   │   │
│ └───────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

#### 7.2.3 关键配置参数

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `enable_blob_files` | false | 开启键值分离 |
| `min_blob_size` | 0 | 小于此值的 value 仍存 SSTable |
| `blob_file_size` | 256MB | Blob 文件目标大小 |
| `enable_blob_garbage_collection` | true | 开启垃圾回收 |
| `blob_garbage_collection_age_cutoff` | 0.25 | GC 时只处理旧文件 |
| `blob_garbage_collection_force_threshold` | 1.0 | 强制 GC 阈值 |

#### 7.2.4 垃圾回收机制

BlobDB 的 GC 在 Compaction 过程中自动进行：

1. **扫描**: 遍历 SSTable 中的 Blob Index
2. **验证**: 检查每个 value 是否仍然有效
3. **复制**: 将有效 value 追加到新 Blob 文件
4. **更新**: 修改 SSTable 中的 Blob Index 指向
5. **清理**: 删除旧 Blob 文件

**GC 触发条件**:
- 配置为定期运行
- 无效数据比例超过阈值
- Compaction 时自动检查

#### 7.2.5 与 WiscKey 的差异

| 特性 | WiscKey | BlobDB |
|------|---------|--------|
| **实现** | 独立系统 | RocksDB 内置 |
| **GC** | 需单独实现 | 与 Compaction 协同 |
| **配置** | 有限 | 丰富的配置选项 |
| **监控** | 基础 | 完善的统计信息 |
| **适用场景** | 大 value 专用 | 通用，自动选择 |

**适用场景**:
- Value 平均大小 > 1KB
- 写入密集，需要降低写放大
- 可以接受轻微的范围查询性能下降


### 7.3 Compaction 策略详解

RocksDB 提供三种主要的 Compaction 策略，适用于不同的工作负载场景。

#### 7.3.1 Leveled Compaction

**源码**: `db/compaction/compaction_picker_level.cc`

**核心思想**: 每层 SSTable 的 key 范围完全不重叠，类似 LevelDB。

**触发条件**:
1. L0 文件数超过 `level0_file_num_compaction_trigger` (默认 4)
2. 某层总大小超过 `max_bytes_for_level_base × (level_multiplier ^ (L-1))`
3. 手动触发 (CompactRange)

**Compaction 选择算法**:
```
1. 优先处理 L0 (如果文件数超过阈值)
2. 否则选择 "最拥挤" 的层 (当前大小 / 目标大小 比例最大)
3. 选择该层中最早创建的 SSTable 作为输入
4. 在下一层找到所有重叠的 SSTables
5. 检查与祖父层的重叠，限制 Compaction 大小
```

**写放大**: 10-30× (取决于数据分布)
**读放大**: 低 (O(1)，每层最多一个 SSTable)

**适用场景**:
- 读多写少 (读/写 > 10:1)
- 范围查询频繁
- 需要稳定读性能

#### 7.3.2 Universal Compaction (Size-Tiered)

**源码**: `db/compaction/compaction_picker_universal.cc`

**核心思想**: 延迟合并，每层允许有多个重叠的 sorted runs，按大小排序。

**触发条件**:
1. 空间放大超过阈值 (`max_size_amplification_percent`)
2. 相近大小的 runs 数量超过阈值
3. 总 run 数超过 `level0_file_num_compaction_trigger`

**合并策略**:
```
1. 空间放大合并: 当总大小 > (100 + 阈值)% 有效数据时
2. 大小比率合并: 合并相近大小的 runs
3. 文件数合并: 当总 run 数过多时
```

**写放大**: 2-5× (显著低于 Leveled)
**读放大**: 较高 (需要检查多个 runs)

**适用场景**:
- 写多读少 (写/读 > 10:1)
- 日志类工作负载
- 顺序写入

#### 7.3.3 FIFO Compaction

**源码**: `db/compaction/compaction_picker_fifo.cc`

**核心思想**: 只保留最新的数据，类似于缓存。

**触发条件**:
- 总大小超过 `compaction_options_fifo.max_table_files_size`
- 或 TTL 过期

**Compaction 行为**:
```
1. 按文件创建时间排序
2. 删除最旧的文件直到大小符合限制
3. 不执行合并操作
```

**写放大**: 极低 (~1×，无重写)
**读放大**: 极高 (可能需要检查大量文件)

**适用场景**:
- 时序数据
- 缓存
- TTL 主导的场景

#### 7.3.4 策略对比总结

| 策略 | 写放大 | 读放大 | 空间放大 | 适用场景 | RocksDB 推荐 |
|------|--------|--------|----------|----------|-------------|
| **Leveled** | 10-30× | 低 (O(1)) | 低 | 读多写少 | 默认 |
| **Universal** | 2-5× | 中-高 | 中 | 写多读少 | 日志类 |
| **FIFO** | ~1× | 极高 | 极高 | 时序/缓存 | 特殊场景 |
| **BlobDB** | 2-5× | 中 | 低 | 大 value | 通用大 value |

- 逻辑隔离不同数据集合
- 每列族独立配置 Compaction 策略
- 支持跨列族原子操作

**9. 监控与诊断**
- 实时统计 (Statistics): 操作计数、耗时分布
- 性能上下文 (PerfContext): 单次操作详细耗时
- 追踪 (Trace): 记录操作序列用于诊断

**10. 写入限速与流控**
- 防止写入过快导致 Compaction 跟不上
- 自动降低写入速率，维持系统稳定
- 可配置的延迟写入策略

## 八、LSM-Tree 优化技术图谱

LSM-Tree 的三大核心性能指标：**写放大**、**读放大**、**空间放大**。本章系统梳理针对这三大问题的完整优化技术体系。

```
LSM-Tree 三大放大问题根源:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  写放大 (Write Amplification)                                │
│  ├── 根源：Compaction 时反复重写数据                         │
│  ├── 影响：降低写入吞吐，缩短 SSD 寿命                        │
│  └── 典型值：LevelDB 12-50×, RocksDB 10-30×                 │
│                                                              │
│  读放大 (Read Amplification)                                 │
│  ├── 根源：需检查多层/多文件才能找到目标 key                  │
│  ├── 影响：增加读取延迟，降低随机读性能                       │
│  └── 典型值：LevelDB 3-14×, RocksDB 2-10×                   │
│                                                              │
│  空间放大 (Space Amplification)                              │
│  ├── 根源：无效数据未及时清理，压缩效率低                     │
│  ├── 影响：浪费磁盘空间，增加成本                            │
│  └── 典型值：LevelDB 1.2×, RocksDB 1.15×                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

---

### 8.1 写放大优化

写放大是 LSM-Tree 最核心的性能问题，定义为**实际写入磁盘的数据量与用户写入数据量的比值**。

#### 8.1.1 写放大来源分析

```
写放大的三大来源:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  1. Compaction 重写                                          │
│     ┌────────────────────────────────────────────────────┐  │
│     │ L0 → L1 Compaction 示例:                            │  │
│     │                                                      │  │
│     │ L0: [File_A][File_B][File_C][File_D]               │  │
│     │              ↓ 合并                                  │  │
│     │ L1: [File_1][File_2][File_3]...[File_N]            │  │
│     │                                                      │  │
│     │ 输入：4 个文件 × 2MB = 8MB                           │  │
│     │ 输出：10 个文件 × 2MB = 20MB                         │  │
│     │ 写放大：20MB / 8MB = 2.5× (仅这一层)                │  │
│     └────────────────────────────────────────────────────┘  │
│                                                              │
│  2. Bloom Filter 更新                                        │
│     - 每次 Compaction 需重新计算 Bloom Filter                │
│     - 额外写入开销：约 5-10%                                 │
│                                                              │
│  3. 索引重建                                                 │
│     - Index Block 和 MetaIndex Block 重建                    │
│     - 额外写入开销：约 3-5%                                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘

总体写放大计算公式:
Write_Amp = Σ(Level_i_Size / Level_i-1_Size) × Compaction_Overhead
```

#### 8.1.2 键值分离优化

**核心思想**: Compaction 只关心 Key 的顺序，Value 可以单独存储，避免反复重写大 Value。

```
键值分离架构对比:

传统 LSM:                           键值分离 LSM:
┌──────────┐                       ┌──────────┐    ┌──────────┐
│ LSM-Tree │                       │ LSM-Tree │    │   vLog   │
│ Key+Value│                       │ Key+Addr │    │  (Value) │
└──────────┘                       └────┬─────┘    └────┬─────┘
       ▲                                │               │
       │                                └───────────────┘
  写放大 12-50×                        写放大 ~1×      顺序追加

优势:
1. Compaction 只处理 Key（通常 16-64 bytes）
2. Value 直接追加到 vLog，不移动
3. GC 后台回收无效 Value，不影响前台写入
```

**代表实现**:

| 系统 | 提出时间 | 关键特性 | 写放大降低 |
|------|----------|----------|------------|
| **WiscKey** | 2016 (FAST) | 学术首创，vLog + 在线 GC | 12× → 1.2× (90%) |
| **Titan** | 2018 | TiKV 引擎，离线 GC | 15× → 1.5× (90%) |
| **BlobDB** | 2020 | RocksDB 集成，生产就绪 | 15× → 2× (87%) |

**WiscKey 详细设计**:

```cpp
// vLog 格式
struct LogEntry {
  uint32_t key_length;
  char[] key_data;
  uint32_t value_length;
  char[] value_data;
  uint32_t checksum;
};

// LSM-Tree 中存储的地址
struct BlobAddress {
  uint64_t file_number;    // vLog 文件编号
  uint64_t offset;         // 在 vLog 中的偏移
  uint32_t size;           // Value 大小
};

// GC 过程
void GarbageCollection() {
  while (true) {
    // 1. 从 Tail 顺序读取 chunk
    entries = ReadChunk(tail_file);
    
    // 2. 验证每个 Key-Value 是否仍有效
    valid_entries = [];
    for (entry : entries) {
      if (LSM.Lookup(entry.key) == entry.address) {
        valid_entries.push(entry);  // 仍被引用
      }
    }
    
    // 3. 有效数据追加写入 Head
    for (entry : valid_entries) {
      new_addr = AppendToHead(entry.key, entry.value);
      LSM.Update(entry.key, new_addr);
    }
    
    // 4. 更新 Tail，释放空间
    tail_file = next_file();
  }
}
```

**Bloom Filter 效果量化**:

| bits_per_key | 误判率 | 内存开销 (每百万 key) | 减少的磁盘读取 |
|--------------|--------|----------------------|----------------|
| 5 | 5.0% | 0.625 MB | 85% |
| 8 | 1.0% | 1.0 MB | 90% |
| 10 | 0.8% | 1.25 MB | 92% |
| 15 | 0.05% | 1.875 MB | 94.5% |

**误判率计算公式**:
```
P(false positive) ≈ (1 - e^(-k*n/m))^k

其中:
- n = key 数量
- m = bits 总数
- k = 哈希函数个数 ≈ 0.693 * (m/n)
- bits_per_key = m/n

LevelDB/RocksDB 默认 bits_per_key = 10
→ k ≈ 7 次哈希
→ 误判率 ≈ 0.8%
```

#### 8.1.3 数据结构优化

**PebblesDB FLSM (Fragmented LSM)**:

```
FLSM vs 传统 LSM 对比:

传统 LSM (Disjoint SSTables):
Level 1: [1-100][200-300]  ← key 范围不重叠
              ↑
         Compaction 时必须重写整个范围

FLSM (允许重叠):
Level 1: Guard:50 ──── Guard:200
         │                │
      [1-60]           [150-250]
      [20-80]          [180-300]
      [40-100]         [200-350]
      ↑ 允许重叠
      Compaction 时仅分区，不重写!

Guard 机制:
- Guard: 从插入 key 中随机选择的划分点
- Guard 之间 disjoint，Guard 内 sstables 可重叠
- 高层包含所有低层 Guards (类似 Skip List)
```

**FLSM Compaction 流程**:

```
传统 LSM Compaction:              FLSM Compaction:
1. 读取 sstable                   1. 选择 Guard（sstables 超过阈值）
2. 合并排序                       2. 按下层 Guards 分区
3. 写入新 sstables                3. 直接附加（不排序！）
4. 删除旧文件                     4. 删除旧文件

→ 数据重写！                      → 仅分区，不重写！

写放大对比:
- 传统 LSM: 18.7×
- PebblesDB (FLSM): 3.9× (降低 79%)
```

**其他数据结构优化**:

| 技术 | 原理 | 写放大降低 | 代表系统 |
|------|------|------------|----------|
| **LSM-trie** | Trie 结构减少重叠 | 15× → 6× (60%) | 研究原型 |
| **Dostoevsky** | 分段 LSM，延迟合并 | 20× → 8× (60%) | 研究原型 |
| **HashKV** | 哈希分区，局部 Compaction | 18× → 7× (61%) | 研究原型 |

#### 8.1.4 Compaction 策略优化

**四种主流 Compaction 策略对比**:

```
1. Leveled Compaction (LevelDB 默认)

   L0: [A-M] [N-Z]           ← 4 个文件触发
        ↓ 合并
   L1: [A-F] [G-L] [M-R] [S-Z]  ← 输出完全排序

   特点:
   - 每层完全排序，相邻层大小比 10:1
   - 读放大最低 (O(1))
   - 写放大最高 (10-30×)
   - 适用：读多写少场景

2. Tiered Compaction (RocksDB Universal)

   L0: [A-C] [D-F] [G-I] [J-L]  ← 4 个 sorted runs
        ↓ 延迟合并
   L1: [A-L]                     ← 一次性合并

   特点:
   - 每层允许多个 sorted runs
   - 写放大最低 (2-5×)
   - 读放大最高 (O(T), T=runs 数)
   - 适用：写多读少场景

3. Leveled-N (混合策略)

   L0-L2: Tiered 风格          ← 低层延迟合并
   L3-L6: Leveled 风格         ← 高层严格排序

   特点:
   - 平衡读写放大
   - 写放大：5-10×
   - 读放大：2-5×
   - 适用：混合负载

4. FIFO Compaction

   保留最近 N 个文件，超出即删除

   特点:
   - 写放大极低 (~1×)
   - 读放大极高
   - 空间放大极高
   - 适用：时序数据、缓存
```

**Compaction 策略选择决策树**:

```
开始
  │
  ▼
工作负载类型？
  │
  ├── 读主导 (读 > 80%)
  │   └──▶ Leveled Compaction
  │        - 低读放大
  │        - 可接受高写放大
  │
  ├── 写主导 (写 > 80%)
  │   ├──▶ Tiered Compaction
  │   │    - 低写放大
  │   │    - 可接受高读放大
  │   │
  │   └──▶ FLSM (PebblesDB)
  │        - 极低写放大
  │        - 中等读放大
  │
  ├── 混合负载
  │   └──▶ Leveled-N 或调优的 Leveled
  │        - 动态调整参数
  │        - 平衡读写
  │
  └── 时序数据/缓存
      └──▶ FIFO Compaction
           - 只保留最新数据
           - 极简 Compaction
```

### 8.2 读放大优化

读放大定义为**实际读取的数据量与请求数据量的比值**，直接影响读取延迟。

#### 8.2.1 读放大来源分析

**插值搜索 (Interpolation Search, RocksDB v11.0+)**:

```
插值搜索原理:

前提：Key 均匀分布 (如时间戳、自增 ID)

传统二分查找:
  mid = (low + high) / 2
  
插值搜索:
  mid = low + ((target - keys[low]) / (keys[high] - keys[low])) * (high - low)
  
  根据 target 值预测位置，而非固定中点

示例:
Keys: [100, 200, 300, 400, 500, 600, 700, 800, 900, 1000]
Target: 850

二分查找:
  第 1 次：mid = 5 (600) < 850，向右
  第 2 次：mid = 7 (800) < 850，向右
  第 3 次：mid = 8 (900) > 850，向左
  第 4 次：找到 850

插值搜索:
  预测位置 = 0 + ((850-100)/(1000-100)) * 9 = 7.5 ≈ 8
  第 1 次：直接定位到 index 8 (900)
  第 2 次：向左找到 850
  
  仅需 2 次比较！

性能提升:
- 均匀分布 key：减少 30-50% 比较次数
- 非均匀分布：自动回退到二分查找
- 配置：index_block_search_type = kInterpolationSearch
```

### 8.3 空间放大优化

空间放大定义为**实际磁盘占用与有效数据量的比值**，直接影响存储成本。

#### 8.3.1 空间放大来源分析

```
空间放大的三大来源:

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  1. 无效数据堆积                                             │
│     ┌────────────────────────────────────────────────────┐  │
│     │ 场景：频繁 Update/Delete                            │  │
│     │                                                      │  │
│     │ T0: Put(k1, v1)  → SSTable_A                        │  │
│     │ T1: Put(k1, v2)  → SSTable_B (v1 未清理)             │  │
│     │ T2: Put(k1, v3)  → SSTable_C (v1,v2 未清理)          │  │
│     │ T3: Delete(k1)   → SSTable_D (v1,v2,v3 未清理)       │  │
│     │                                                      │  │
│     │ 有效数据：0 (已删除)                                 │  │
│     │ 实际占用：4 个版本 × size                            │  │
│     │ 空间放大：∞ (无穷大)                                 │  │
│     └────────────────────────────────────────────────────┘  │
│                                                              │
│  2. 压缩效率低                                               │
│     - 未启用压缩：1.0× (无节省)                              │
│     - Snappy: 1.3-1.5× (节省 30-40%)                         │
│     - Zstd: 1.1-1.3× (节省 50-60%)                           │
│                                                              │
│  3. 元数据开销                                               │
│     - InternalKey: +8 bytes/key (seq+type)                   │
│     - Bloom Filter: 1.25MB/百万 keys                         │
│     - Index Block: 约 5-10% 额外空间                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘

Space_Amp = (Raw_Data + Metadata + Invalid_Data) / Effective_Data

### 8.4 三大优化技术协同

在实际系统中，写放大、读放大、空间放大相互制约，需要综合权衡。

```
Trade-off 三角关系:

                    写放大
                     / \
                    /   \
                   /     \
                  /       \
                 /         \
                /           \
            读放大 ←────→ 空间放大

典型权衡场景:

1. 降低写放大 ↔ 增加读放大
   - Tiered Compaction: 写放大↓, 读放大↑
   - 键值分离：写放大↓↓, 随机读↑

2. 降低写放大 ↔ 增加空间放大
   - 延迟 Compaction: 写放大↓, 空间放大↑
   - FIFO 策略：写放大↓↓, 空间放大↑↑

3. 降低读放大 ↔ 增加空间放大
   - Bloom Filter: 读放大↓, 内存占用↑
   - 更多缓存：读放大↓, 内存占用↑

最优策略:
根据具体 workload 选择合适的平衡点!
```

**Workload 驱动的优化选择**:

```

├─────────────────────────────────────────────────────────────┤

| **RocksDB** | 2012 | 10-30× | 2-10× | 多线程、丰富特性 | 工业通用 |
| **WiscKey** | 2016 | ~1× | ~1×+1 | 键值分离 | 大 value |
| **PebblesDB** | 2017 | 3-4× | 中 | FLSM | 高吞吐写 |
| **Titan** | 2018 | ~1× | ~1×+1 | 键值分离 | TiKV 生态 |
| **BlobDB** | 2020 | 2-5× | 2-5× | 工业级键值分离 | RocksDB 生态 |

### 9.2 优化效果量化

以 LevelDB 为基准（100%），对比各优化方案的效果：

**写入吞吐（随机写入，MB/s）**:
| 系统 | 吞吐量 | 相对基准 |
|------|--------|----------|
| LevelDB | 10 | 100% |
| RocksDB | 15 | 150% |
| PebblesDB | 27 | 270% |
| WiscKey (4KB value) | 350 | 3500% |
| BlobDB (4KB value) | 400 | 4000% |

**写放大（数值越低越好）**:
| 系统 | 写放大 | 降低比例 |
|------|--------|----------|
| LevelDB | 12× | - |
| RocksDB | 15× | -25% |
| PebblesDB | 4× | 67% |
| WiscKey | 1.2× | 90% |
| BlobDB | 2× | 83% |

**读取性能（随机点查，KOps/s）**:
| 系统 | 点查吞吐 | 相对基准 |
|------|----------|----------|
| LevelDB | 25 | 100% |
| RocksDB | 30 | 120% |
| PebblesDB | 25 | 100% |
| WiscKey | 300 | 1200% |
| BlobDB | 37 | 150% |

**空间效率（100GB 原始数据实际占用）**:
| 系统 | 实际占用 | 空间放大 |
|------|----------|----------|
| LevelDB | 120 GB | 1.2× |
| RocksDB | 115 GB | 1.15× |
| WiscKey (GC 后) | 102 GB | 1.02× |
| BlobDB | 105 GB | 1.05× |

**优化收益总结**:
- **键值分离**是降低写放大最有效的手段（WiscKey/BlobDB）
- **Bloom Filter**是降低读放大最有效的手段（减少 90%+ 无效磁盘读取）
- **压缩算法**选择对空间效率影响显著（Zstd 比 Snappy 节省 30-50% 空间）
- **多线程**可显著提升写入吞吐（RocksDB 比 LevelDB 高 50%+）
